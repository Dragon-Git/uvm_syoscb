5 @section sGeneralStructure General structure
6 
7 Each scoreboard (cl_syoscb) consists of a number of queues (see \ref pQueueImplementationNotes).
8 A scoreboard should have as many queues as there are models being tested.
9 If e.g. the testbench compares a DUT to a single reference model, two queues, "DUT" and "REF" should be instantiated.
10 Each queue is associated with a number of producers. If e.g. the inputs and outputs of the DUT/REF are being sampled,
11 these producers may be named "IN" and "OUT".
12 
13 The scoreboard operates by using TLM connections or the function-based API to receive data sampled by a UVM monitor or a VC.
14 The scoreboard has a subscriber for each producer on each queue. When a sequence item is written to a subscriber,
15 the sequence item is wrapped in a cl_syoscb_item before it is added to the correct queue.
16 The cl_syoscb_item is a wrapper item which includes some metadata such as which producer generated the item.
17 
18 The queues in the scoreboard store sequence items until at least one sequence item is present in each queue.
19 When this condition is met, a comparison is started, based on the chosen compare algorithm (see \ref pCompareImplementationNotes).
20 Every time an item is inserted during simulation, the scoreboard checks if all queues contain an item,
21 and starts a comparison if this is the case. It does so throughout the entire simulation.
22 If any errors occur, the scoreboard will generate a UVM_ERROR.
23 See \ref pDebugFeatures for more information on the debugging features that the scoreboard includes.
24 
25 Note that when using the SyoSil UVM Scoreboard, the reference model and scoreboard are completely separated,
26 which is not the case in many other UVM scoreboards.
27 By separating scoreboard and reference model, we achieve separation of concerns, making both the reference model and scoreboard simpler to use and instantiate in other testbenches.

38 @section sGeneralErrorHandling General error handling
39 Whenever a method detects an error, two error handling concepts are used:
40 
41 Getter methods and other methods with a return value will generally issue a UVM_INFO message at verbosity level UVM_DEBUG with some information about what went wrong, returning \c 1'b0 or \c null to signal an error. It is up to the calling method to respond to and handle the error, or escalate it to a UVM_ERROR / UVM_FATAL.
42 
43 Setter methods and other methods without a return value will issue a UVM_ERROR or UVM_FATAL, as they have no other way of signalling errors. Again, it is the responsibility of the caller to potentially catch and handle these errors.

83 @section sMultipleQueueRefs Multiple queue references
84 Both the top level scoreboard class cl_syoscb and the configuration class cl_syoscb_cfg contain
85 handles to all queues. The former uses an ordinary array which provides a fast way of
86 looping over the queues and the latter an associative which makes it easy to find
87 a queue using only its name. Using either reference is OK, use the approach best suited
88 for a given operation.
